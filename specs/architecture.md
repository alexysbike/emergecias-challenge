# Architecture

Contact agenda API. Lightweight Clean Architecture with a class-based HTTP abstraction, Drizzle ORM (SQLite), Zod validation, centralized errors, OpenAPI generation on startup, Jest tests, and ESLint/Prettier.

Related specs: [models.md](./models.md), [endpoints.md](./endpoints.md).

---

## Principles

| Principle | Application |
|-----------|-------------|
| Dependency rule | Inner layers never import from outer layers |
| Light Clean Architecture | Use cases + repository ports; no event bus, no CQRS |
| Single validation source | Zod schemas drive runtime validation and OpenAPI |
| Framework at the edge | Express lives only in `infrastructure/http/express` |
| Fail explicitly | Use cases throw typed errors; HTTP layer maps them to responses |
| Class-based routing | Each endpoint is an `HttpRoute` subclass; routers group routes |

## Layer responsibilities

### Domain

- Pure TypeScript: entities and repository **ports** (interfaces).
- No Express, Drizzle, or Zod imports.
- Entities mirror [models.md](./models.md).

Example port:

```typescript
// domain/repositories/contact.repository.ts
export interface ContactRepository {
  create(input: CreateContactInput): Promise<Contact>;
  findById(id: number): Promise<Contact | null>;
  findByEmail(email: string): Promise<Contact | null>;
  list(filters: ContactListFilters): Promise<Paginated<Contact>>;
  update(id: number, input: UpdateContactInput): Promise<Contact>;
  delete(id: number): Promise<void>;
}
```

### Application

- One use case per business operation (aligned with [endpoints.md](./endpoints.md)).
- Depends on repository ports and throws `AppError` or shortcut subclasses.
- Input/output DTOs; no HTTP types.

Example flow вҖ” `CreateContactUseCase`:

1. Check email uniqueness вҶ’ `ConflictError`
2. Validate `phoneTypeId` exists вҶ’ `ValidationError`
3. Persist via `ContactRepository.create`
4. Return `Contact` entity

### Infrastructure

- **Drizzle**: schema, migrations, repository implementations.
- **Express adapter**: implements `HttpServer` only here.
- **Seed**: `PhoneType` rows (`Mobile`, `Home`, `Work`) via seed script or migration.

### Presentation

- Each endpoint is an `HttpRoute` subclass.
- Routers (`HttpRouter` subclasses) group related routes.
- Zod schemas live in `presentation/validators/`; reused by routes and OpenAPI registry.
- Mappers convert domain entities to API response shapes from [endpoints.md](./endpoints.md).

### Composition root

- `createApp()` / `createHttpServer()` builds: DB client вҶ’ repositories вҶ’ use cases вҶ’ routes вҶ’ `HttpServer`.
- Single place to swap Express for another adapter in tests or future migration.

## HTTP abstraction

Express is isolated behind a class-based abstraction. Use cases and routes never import `express`.

### HttpMethod enum

```typescript
// infrastructure/http/http-method.ts
export enum HttpMethod {
  GET = "GET",
  POST = "POST",
  PUT = "PUT",
  PATCH = "PATCH",
  DELETE = "DELETE",
}
```

### HttpRequest / HttpResponse

Framework-agnostic request/response interfaces:

```typescript
export interface HttpRequest {
  method: HttpMethod;
  path: string;
  params: Record<string, string>;
  query: Record<string, string | string[] | undefined>;
  body: unknown;
}

export interface HttpResponse {
  status(code: number): HttpResponse;
  json(body: unknown): void;
  send(): void; // 204 No Content
}
```

`HttpResponse` is used only in infrastructure (Express adapter, middleware, error handler). Route handlers never receive it.

### HttpResult

Route handlers return a response body, an explicit `HttpResult`, or nothing. `HttpRoute.execute()` normalizes the return value and writes to `HttpResponse`.

```typescript
// infrastructure/http/http-result.ts
export class HttpResult {
  static ok(body: unknown): HttpResult;
  static created(body: unknown): HttpResult;
  static noContent(): HttpResult;
  static of(status: number, body?: unknown): HttpResult;
}

export type RouteHandlerReturn = unknown | HttpResult | void;
```

**Default status by HTTP method** (when the handler returns a plain body or `void`):

| Method | Return value | Implicit status |
|--------|--------------|-----------------|
| `GET` | body | `200` |
| `POST` | body | `201` |
| `PUT` / `PATCH` | body | `200` |
| `DELETE` | `void` / `undefined` | `204` (no body) |

Use `HttpResult` when the status must differ from the default (e.g. `POST` вҶ’ `200`, custom codes).

### HttpRoute (abstract)

Each route declares its path, allowed methods, and handler logic via overridable verb methods. Handlers return data; they do not call `res.status().json()`.

```typescript
// infrastructure/http/http-route.ts
export abstract class HttpRoute {
  abstract readonly path: string;
  abstract readonly methods: HttpMethod[];

  /** Entry point: validates method, delegates, normalizes return, writes response */
  async execute(req: HttpRequest, res: HttpResponse): Promise<void> {
    if (!this.methods.includes(req.method)) {
      throw new AppError(
        `Method ${req.method} not allowed`,
        405,
        ErrorCode.METHOD_NOT_ALLOWED
      );
    }

    const raw = await this.dispatch(req);
    const result = this.normalizeResult(req.method, raw);
    this.sendResult(res, result);
  }

  protected get(_req: HttpRequest): Promise<RouteHandlerReturn> | RouteHandlerReturn {
    throw new AppError("GET not implemented", 405, ErrorCode.METHOD_NOT_ALLOWED);
  }
  protected post(_req: HttpRequest): Promise<RouteHandlerReturn> | RouteHandlerReturn {
    throw new AppError("POST not implemented", 405, ErrorCode.METHOD_NOT_ALLOWED);
  }
  protected put(_req: HttpRequest): Promise<RouteHandlerReturn> | RouteHandlerReturn {
    throw new AppError("PUT not implemented", 405, ErrorCode.METHOD_NOT_ALLOWED);
  }
  protected patch(_req: HttpRequest): Promise<RouteHandlerReturn> | RouteHandlerReturn {
    throw new AppError("PATCH not implemented", 405, ErrorCode.METHOD_NOT_ALLOWED);
  }
  protected delete(_req: HttpRequest): Promise<RouteHandlerReturn> | RouteHandlerReturn {
    throw new AppError("DELETE not implemented", 405, ErrorCode.METHOD_NOT_ALLOWED);
  }
}
```

Subclasses override only the verb methods listed in `methods`.

### HttpRouter (abstract)

Groups `HttpRoute` instances under a path prefix. Routers can nest other routers via `mount()`. Route classes declare **relative** paths; the full URL is built when the tree is flattened.

```typescript
// infrastructure/http/http-router.ts
export abstract class HttpRouter {
  abstract readonly path: string; // segment, e.g. "/contacts" or "/:personId/activities"
  protected readonly routes: HttpRoute[] = [];
  protected readonly routers: HttpRouter[] = [];

  register(...routes: HttpRoute[]): this {
    this.routes.push(...routes);
    return this;
  }

  mount(...routers: HttpRouter[]): this {
    this.routers.push(...routers);
    return this;
  }

  getRoutes(): readonly HttpRoute[] {
    return this.routes;
  }

  getRouters(): readonly HttpRouter[] {
    return this.routers;
  }
}
```

**Flattening nested routers** (`path-utils.ts`):

```typescript
export function flattenRouters(

  // joins basePath + router.path + route.path recursively

```

### HttpServer

Receives configuration in the constructor; starts the server via `run()`.

```typescript
// infrastructure/http/http-server.config.ts
export interface HttpServerConfig {
  port: number;
  routers: HttpRouter[];
  middlewares?: HttpMiddleware[];
  env: Env;
}

// infrastructure/http/http-server.ts
export abstract class HttpServer {
  constructor(protected readonly config: HttpServerConfig) {}

  abstract run(): void;
}
```

### Express adapter

`ExpressHttpServer` extends `HttpServer` and is the only class that imports Express.

On `run()`:

1. Apply global middlewares (JSON parser, etc.).
2. Flatten routers and routes; register each resolved path + methods on Express.
3. Wrap each handler: adapt Express req/res вҶ’ call `route.execute()`.
4. Register centralized error handler as last middleware.
5. Mount Swagger UI at `/doc` when not in production.
6. Listen on configured port.

```typescript
// infrastructure/http/express/express-http-server.ts
export class ExpressHttpServer extends HttpServer {
  private readonly app = express();

  run(): void {
    this.config.middlewares?.forEach((mw) => this.app.use(wrapMiddleware(mw)));

    for (const { path, route } of flattenRouters(this.config.routers)) {
      const handler = asyncHandler(async (req, res) => {
        await route.execute(adaptRequest(req), adaptResponse(res));
      });

      for (const method of route.methods) {
        this.app[toExpressMethod(method)](path, handler);
      }
    }

    this.app.use(wrapErrorHandler(errorHandler));
    registerSwaggerUi(this.app, this.config.env);
    this.app.listen(this.config.port, onListen);
  }
```

## Error handling

### Error response contract

From [endpoints.md](./endpoints.md):

```json
{
  "error": "Human-readable error description",
  "code": "VALIDATION_ERROR"
}
```

Suggested codes: `VALIDATION_ERROR`, `NOT_FOUND`, `CONFLICT`, `INTERNAL_ERROR`, `METHOD_NOT_ALLOWED`.

### Standard AppError

The core error type accepts `message`, `statusCode`, and `code` in the constructor. Shortcut subclasses are optional conveniences for common cases.

```typescript
// shared/errors/error-codes.ts
export const ErrorCode = {
  VALIDATION_ERROR: "VALIDATION_ERROR",
  NOT_FOUND: "NOT_FOUND",
  CONFLICT: "CONFLICT",
  INTERNAL_ERROR: "INTERNAL_ERROR",
  METHOD_NOT_ALLOWED: "METHOD_NOT_ALLOWED",
} as const;

export type ErrorCode = (typeof ErrorCode)[keyof typeof ErrorCode];

// shared/errors/app.error.ts
export class AppError extends Error {
  constructor(
    message: string,
    public readonly statusCode: number,
    public readonly code: ErrorCode
  ) {
    super(message);
    this.name = "AppError";
  }

  toJSON(): { error: string; code: ErrorCode } {
    return { error: this.message, code: this.code };
  }
}
```

**Shortcut subclasses:**

```typescript
export class ValidationError extends AppError {
  constructor(message: string) {
    super(message, 400, ErrorCode.VALIDATION_ERROR);
  }
}

export class NotFoundError extends AppError {
  constructor(message = "Resource not found") {
    super(message, 404, ErrorCode.NOT_FOUND);
  }
}

export class ConflictError extends AppError {
  constructor(message: string) {
    super(message, 409, ErrorCode.CONFLICT);
  }
}

export class InternalError extends AppError {
  constructor(message = "Internal server error") {
    super(message, 500, ErrorCode.INTERNAL_ERROR);
  }
}
```

**Direct usage** when a custom status is needed:

```typescript
throw new AppError("Search filters are mutually exclusive", 400, ErrorCode.VALIDATION_ERROR);
```

| Rule | Detail |
|------|--------|
| Prefer shortcuts | For standard codes from [endpoints.md](./endpoints.md): `400`, `404`, `409`, `500` |
| Use `AppError` directly | When status or code does not fit a shortcut |
| Use cases | Throw errors only; never set HTTP status codes |

### Centralized error handler

```typescript
// infrastructure/http/express/middleware/error-handler.middleware.ts
export function errorHandler(err: unknown, _req: HttpRequest, res: HttpResponse): void {
  if (err instanceof AppError) {
    res.status(err.statusCode).json(err.toJSON());
    return;
  }
  if (err instanceof ZodError) {
    res.status(400).json({
      error: err.errors[0]?.message ?? "Validation failed",
      code: ErrorCode.VALIDATION_ERROR,
    });
    return;
  }
  console.error(err);
  res.status(500).json(new InternalError().toJSON());
}
```

Registered as the last middleware on the Express app inside `ExpressHttpServer.run()`.
